# AI-DLC End of Day（作業終了記録）

1日の作業内容を収集・記録し、AI-DLCワークフローの監査証跡を維持する。

## 実行手順

### Step 1: aidlc-docsの存在確認

`aidlc-docs/audit.md` と `aidlc-docs/aidlc-state.md` が存在するか確認する。存在しない場合はユーザーに通知して終了する。

### Step 2: 作業内容の収集

以下の3つのソースから本日の作業内容を収集する：

**2a. git差分の収集（記録済みの除外）**

まず `aidlc-docs/audit.md` を読み、既に記録済みのコミットハッシュやTimestampを把握する。

```bash
# 本日の日付を取得（YYYY-MM-DD形式）
today=$(date +%Y-%m-%d)

# 本日のコミット一覧（あれば）
git log --oneline --after="${today}T00:00:00" --before="${today}T23:59:59"

# 未コミットの変更（ステージ済み＋未ステージ）
git diff --stat
git diff --cached --stat

# 変更されたファイルの一覧
git diff --name-only HEAD
```

**重複除外ルール**:
- audit.mdの既存エントリのTimestamp・作業タイトル・変更ファイルリストを読み取る
- audit.mdの最後のエントリのTimestampより**後**に行われた変更のみを記録対象とする
- audit.mdの最後のエントリに既に記載されている変更内容と明らかに同一の作業は記録対象外とする
- 未コミットの変更は常に記録対象とする
- 判定に迷う場合は記録する側に倒す（漏れよりも重複の方が害が小さい）

**2b. 変更ファイルの分類**
収集した差分を以下のカテゴリに分類する：
- **コード変更**: src/ 配下のファイル
- **設計書変更**: aidlc-docs/ 配下のファイル
- **設定変更**: config、package.json等
- **テスト変更**: tests/ 配下のファイル
- **その他**: 上記以外

**2c. 会話コンテキストの活用**
git差分だけでは「なぜ」その変更をしたかが分からない。会話の文脈から以下を補完する：
- 変更の動機・ユーザーの指摘内容
- 決定の理由
- 試行錯誤の経緯（複数回の調整があった場合）

### Step 3: audit.md への記録追記

`aidlc-docs/audit.md` の末尾に以下の形式で追記する：

```markdown
## [作業タイトル（簡潔に内容を表す）]
**Timestamp**: [ISO 8601形式の日時範囲]
**User Input**: [ユーザーの指示・指摘の要約]
**AI Response**: 以下の変更を実施。

### コード変更
[番号付きリストで変更内容を列挙。各項目は「何を」「どう変えたか」を簡潔に]

### 設計書更新
[更新した設計書ファイルと変更箇所]

### 変更ファイル
[変更されたファイルのリスト（パスと変更概要）]

**決定理由**: [なぜこの変更を行ったかの要約]
**Context**: [AI-DLCフェーズ・ステージの文脈]

---
```

記録のポイント：
- 変更の「何を」「なぜ」を必ず含める
- 複数の独立した作業があった場合はセクションを分ける
- 試行錯誤の経緯は省略し、最終的な結果を記録する

### Step 4: aidlc-state.md の更新

`aidlc-docs/aidlc-state.md` を読み、以下を確認・更新する：
- **Current Stage** の記述が現状と合っているか
- ステージの進捗チェックボックスが最新か
- 新たに完了したステージがあればチェックを入れる

### Step 5: 未コミット変更の通知

以下を確認してユーザーに報告する：

```bash
git status --short
```

- 未コミットの変更がある場合：変更ファイル一覧を表示し、コミットするか確認する
- 変更がない場合：「全ての変更はコミット済みです」と報告する

### Step 6: サマリの表示

最後にユーザーへ以下のサマリを表示する：
- audit.mdに記録した作業内容の要約（箇条書き）
- aidlc-state.mdの更新内容（あれば）
- 未コミット変更の有無
