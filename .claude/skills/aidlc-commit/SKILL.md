# AI-DLC Commit（設計書整合チェック付きコミット）

コミット前に設計書との整合性を確認し、必要に応じて設計書を更新してからコミットする。

## 実行手順

### Step 1: 変更内容の把握

```bash
git status
git diff --cached --stat   # ステージ済みの変更
git diff --stat            # 未ステージの変更
```

**コミット対象の判定**:
- ステージ済みの変更がある場合 → ステージ済みの変更のみをコミット対象とする
- ステージ済みの変更がなく、未ステージの変更がある場合 → 未ステージの変更一覧を表示し「これらをステージしてコミットしますか？」とユーザーに確認する
- どちらもない場合 → 「コミットする変更がありません」と通知して終了する

### Step 2: AI-DLCプロジェクト判定

`aidlc-docs/` ディレクトリが存在するか確認する。
- 存在しない場合 → 通常のコミットフロー（Step 5へスキップ）に進む
- 存在する場合 → Step 3 の設計書チェックへ進む

### Step 3: 設計書との整合性チェック

`src/` 配下に変更があるファイルを抽出し、以下の設計書と照合する。

**照合対象の設計書**:
- `aidlc-docs/**/domain-entities.md` — エンティティ定義（サイズ、初期値、コライダー等の数値パラメータ）
- `aidlc-docs/**/business-rules.md` — ビジネスルール（移動制限、射撃方向、回収半径等のゲームルール）
- `aidlc-docs/**/business-logic-model.md` — ビジネスロジック（アルゴリズム、処理フロー、数値定義）

**チェック方法**:

変更された `src/` ファイルの `git diff` を読み、以下に該当する変更がないか判定する：

1. **数値パラメータの変更** — config値、サイズ、半径、速度、間隔などの定数が変わった場合、設計書に同じ値の記載がないか確認する
2. **ロジックの変更** — アルゴリズムや処理フローが変わった場合、設計書に該当するロジック記述がないか確認する
3. **新規機能の追加** — 設計書に記載のない新しい仕組みが追加された場合
4. **機能の削除・無効化** — 設計書に記載されている機能が削除された場合

**判定結果**:
- **更新不要**: 設計書に影響する変更なし → Step 5 へ
- **更新必要**: 影響する箇所を具体的にリストアップ → Step 4 へ

### Step 4: 設計書の更新

更新が必要な場合、ユーザーに以下を提示する：

```
設計書の更新が必要です:

- [ファイル名]: [変更箇所の説明]
  例) domain-entities.md: プレイヤーのcolliderRadius 16→72

設計書を更新してからコミットしますか？
  A) 設計書を更新してコミット（推奨）
  B) 設計書の更新なしでコミット
  C) コミットを中止
```

- A の場合: 設計書を更新し、更新した設計書もコミット対象に含める
- B の場合: そのまま Step 5 へ
- C の場合: 終了

### Step 5: コミットメッセージの提案

以下の手順でコミットメッセージを生成し、ユーザーに提示する。**コミットの実行はユーザー自身が行う。**

1. `git log --oneline -5` で直近のコミットメッセージスタイルを確認
2. 変更内容に基づいてConventional Commits形式の日本語コミットメッセージを生成
3. `git add` すべきファイルのリストと合わせて提示する

**コミットメッセージ形式**:
```
<type>: <日本語の簡潔な説明>
```
type: feat / fix / refactor / docs / chore / style / test

設計書とコードを同時にコミットする場合、コミットメッセージにその旨を含める。
例: `feat: キャラクターサイズ拡大と設計書の同期更新`

**提示フォーマット**:
```
以下のコマンドでコミットできます:

git add [ファイルリスト]
git commit -m "<提案メッセージ>"
```

ユーザーがメッセージの修正を求めた場合は再提案する。
